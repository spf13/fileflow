## 2024-04-23 - Avoid bufio.Writer with io.Copy for os.File
**Learning:** Wrapping a standard file (`*os.File`) in `bufio.Writer` when using `io.Copy` disables zero-copy system calls like `sendfile` or `copy_file_range`. This degrades file copy performance.
**Action:** Use direct `io.Copy` with `*os.File` instances directly, instead of wrapping them in a buffered writer, to take advantage of OS-level zero-copy optimizations.

**Learning:** Using `sync.Pool` to reuse `[]byte` buffers for `io.CopyBuffer` and manual `io.Read` loops significantly reduces memory allocations per operation by eliminating the need to repeatedly allocate large slices (e.g., 32KB). When storing mutable slices in a `sync.Pool` where the expected length might change at runtime based on global config (`BufferSize`), carefully manage capacity: ensure slices are resliced to the target size (`b[:size]`) before being used to prevent reading short or trailing zero bytes from previous uses. Also, store pointers to slices (`*[]byte`) in the pool to avoid additional heap allocations when converting a slice struct into an `interface{}`.
**Action:** When implementing high-throughput file utilities, cache and reuse read/write buffers via `sync.Pool` to cut GC pressure. Always store pointers to slices, verify capacity against required size upon retrieval, and accurately reslice bounds before executing the I/O operation.
